Pre-v1

Rehearse is in active development. The capabilities below are shipped and working today (in the specscore-cli runtime). There is no stable release yet — expect changes, and don't build on it as if the format were frozen.

01 — Real execution

The description is the test. No glue, no mocks.

A scenario is a single Markdown file where human-readable steps are the executable ones. A ```bash, ```sql, ```hurl (HTTP), or ```graphql block under a ## Given / ## When / ## Then heading runs directly — no fixture classes, no step-definition layer, no separate "here's the spec / there's the code" split.

And nothing is faked. Rehearse runs the real commands against a real, throwaway filesystem, one scenario at a time for determinism. A mock rigged to pass proves nothing; real execution is the whole point.

02 — Reuse, not copy-paste

Write a check once. Invoke it with **Use:**.

A check is a reusable, parameterized verification unit — write "the session cookie is hardened" once, then invoke it from every flow that needs it. A scenario calls a check with a **Use:** directive: a plain Markdown link (clickable on GitHub) plus name=value parameters.

session-hardened.check.md — written once
# Check: session-hardened

**Params:** response_file

```bash
grep -qi '^set-cookie: session=' {{response_file}} || { echo "no session cookie"; exit 1; }
grep -i  '^set-cookie: session=' {{response_file}} | grep -qi 'HttpOnly' || { echo "not HttpOnly"; exit 1; }
echo "session hardened"
```
email-login.md, google-login.md, facebook-login.md — each reuses it
## Then the session is hardened
**Use:** [session-hardened](./_checks/session-hardened.check.md) with response_file=out.txt

03 — Markdown-native, agent-authorable

Read on GitHub, run in CI, authored fluently by an AI agent.

Scenarios and checks are plain Markdown in git — reviewable in a pull request, diffable, no proprietary format. The same file a product owner reads, a developer runs, and an AI coding agent writes.

Acceptance criteria stay thin: each lives as a standalone _acs/*.ac.md file stating only what must be true, and specscore rehearse acs generates a readable ## Acceptance Criteria summary from them — single source of truth, one projection to read.

04 — Evidence, not vibes

Rehearse tests itself.

Rehearse's own acceptance criteria are proven by Rehearse scenarios, run in CI. It eats its own cooking: the reuse property above is verified end-to-end by a three-method login demo where email, Google, and Facebook flows each invoke the same two shared checks.

Results export as JSON and feed back into SpecScore as verified-behavior facts — so a spec can give an honest, current answer to "does this still actually work?"
